# Agent Command Center Progress Metric Definitions v0.1

## Current Context
- Station Chief runtime is parked at v4.7.0.
- This is a non-runtime progress metric definition document.
- This document does not modify runtime behavior.
- This document does not create v4.8.
- This document does not authorize runtime behavior.

## Purpose
This document defines what progress percentages and status categories mean when reporting Agent Command Center progress.

- progress metrics are descriptive only
- progress metrics do not grant permission
- progress metrics do not select next tasks
- progress metrics do not authorize runtime work
- progress metrics do not resume Station Chief
- progress metrics do not create v4.8

## Progress Metric Principle
- progress numbers are operator-visibility estimates
- progress numbers are not execution authorization
- high percentage does not mean runtime work may proceed
- low percentage does not mean builder may choose tasks
- progress categories do not override Station Chief parking
- progress reporting does not authorize APIs, network, deployment, production, workers, tasks, queues, routing, or v4.8

## Progress Categories

- **Overall Agent Command Center / Station Chief project**
  - what it measures: Aggregate project delivery state.
  - what it does not measure: Individual runtime execution capability.
  - example evidence: Commit counts, document completion logs.
  - runtime effect: None
  - whether it authorizes future work: No

- **Runtime safety spine**
  - what it measures: Validation suite coverage.
  - what it does not measure: Current system execution stability.
  - example evidence: Validator pass counts.
  - runtime effect: None
  - whether it authorizes future work: No

- **Governance / operating doctrine**
  - what it measures: Protocol/handbook completion.
  - what it does not measure: Operator compliance.
  - example evidence: Documentation file presence.
  - runtime effect: None
  - whether it authorizes future work: No

- **Worker architecture design**
  - what it measures: Family mapping and glossary completion.
  - what it does not measure: Worker capability.
  - example evidence: Architecture map completeness.
  - runtime effect: None
  - whether it authorizes future work: No

- **Controlled local execution capability**
  - what it measures: Artifact generation/Closeout candidate coverage.
  - what it does not measure: Live worker routing capability.
  - example evidence: Successful artifact generation records.
  - runtime effect: None
  - whether it authorizes future work: No

- **Actual live worker/tool automation**
  - what it measures: Live integration capability.
  - what it does not measure: Current runtime enablement status.
  - example evidence: Planned API hook existence.
  - runtime effect: None
  - whether it authorizes future work: No

- **Full command center vision**
  - what it measures: Aggregate architectural completeness.

  - whether it authorizes future work: No

- **Operator authority coverage**

## Progress Band Definitions

- 0–10% — Concept only: Initial draft status. Interpreation: Pure idea. Status: Not active.
- 11–25% — Early structure: Skeleton documents created. Interpretation: Early draft. Status: Not active.
- 26–50% — Partial framework: Core concepts mapped. Interpretation: Preliminary. Status: Not active.
- 51–70% — Functional planning base: Foundation present. Interpretation: Functional planning. Status: Not active.
- 71–85% — Strong documented structure: Most content defined. Interpretation: Solid base. Status: Not active.
- 86–95% — Mature planning/governance structure: Final review pending. Interpretation: Near completion. Status: Not active.
- 96–99% — Near-complete non-runtime framework: Minor tweaks pending. Interpretation: Awaiting landing. Status: Not active.
- 100% — Complete only for the defined scope: Fully specified. Interpretation: Finished scope. Status: Not active.

## Runtime vs Non-Runtime Metric Split

- **Runtime progress**: Building/testing runtime layers.
- **Non-runtime documentation progress**: Drafting files in `09_exports/`.
- **Governance progress**: Drafting authority/protocol files.
- **Worker architecture design progress**: Mapping worker families.
- **Live automation progress**: Planning API/network interfaces.
- **Production readiness progress**: Defining production candidates.

For each:
- what counts: Completion of documentation files for that area.
- what does not count: Runtime execution/deployment.
- what approvals are still required: Runtime layer tokens.

## Metric Misinterpretation Warnings

- high governance progress does not authorize v4.8
- high documentation coverage does not authorize runtime work
- high worker architecture design does not start workers
- high dashboard readiness does not create dashboard automation
- high local execution capability does not authorize production
- live automation percentage must remain low until explicit external action approval
- production readiness is not production authorization
